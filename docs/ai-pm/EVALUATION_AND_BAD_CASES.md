# 评测、指标与 Bad Case 闭环

## 1. 当前离线评测

运行：

```bash
python src/evaluate_copilot.py
```

`eval/golden_cases.csv` 是 24 条合成金标回归集，覆盖六类常规异常；`eval/bad_cases.csv` 是 6 条证据缺失或互相矛盾的安全测试。它们用于防止代码迭代破坏既有规则，**不能替代真实线上效果**。

当前自动化检查：

| 层级 | 指标 | 定义 | 首版门槛 |
|---|---|---|---|
| 规则 | 裁定准确率 | 裁定与金标一致 | 100% 通过回归集 |
| 规则 | 建议动作准确率 | 动作与金标一致 | 100% 通过回归集 |
| 安全 | 安全兜底通过率 | 证据不足时进入人工复核 | 100% |
| 业务 | 人工采纳率 | 采纳建议 / 已审核建议 | 观察，不预设虚构目标 |
| 效率 | 每案处理时长 | 提交决定时间 - 打开案件时间 | 与人工基线对比 |
| 系统 | P95 时延、单案成本 | API/模型调用指标 | 运行中采集 |

## 2. 上线前评测计划

1. 从真实历史账单随机抽样，按异常类型、金额区间和复杂度分层；每类先收集 100–300 条。  
2. 由两名结算专家独立标注异常类型、证据、建议动作和“是否必须升级人工”，对分歧裁决并记录版本。  
3. 将评测拆为“证据召回—规则裁定—建议动作—人工采纳”四层，避免只看最终准确率。  
4. 保留冻结测试集；提示词、模型或规则升级只能在开发集调优，最终在冻结集复测。  
5. 上线后把驳回、升级和重复追问回流为候选 Bad Case，每周按风险金额和发生频率排序复盘。

## 3. Bad Case 分类与处置

| Bad Case | 风险 | 兜底 | 后续优化 |
|---|---|---|---|
| 订单或费用明细缺失 | 证据不足却给出资金建议 | `SUSPECT + 人工复核` | 补数据血缘和缺失字段提示 |
| 重复计费与账单行数冲突 | 聚合/主数据错误 | 不输出拒付建议 | 排查账单粒度与去重键 |
| 合同条款版本不明 | 错用过期容差 | 人工复核并标记策略版本 | 建立版本化条款库 |
| 跨币种、税费或舍入差异 | 金额比较失真 | 人工复核 | 增加币种、税率和舍入规则 |
| 模型理由与结构化证据不一致 | 幻觉或过度解释 | 只展示规则理由，屏蔽模型建议 | 强制 JSON Schema 与证据引用校验 |
| 高金额、低置信度案件 | 错误成本高 | 强制主管升级 | 金额阈值与双人审批 |

## 4. 不夸大的结果表述

本仓库可表述为：“在合成账单与注入真值上，对账引擎异常查全率约 99.9%；新增的 Copilot 回归集和安全兜底测试用于验证规则一致性。”

不可表述为：“AI 复核准确率 99.9%”或“自动追回 14 万元”。14 万元是模拟账单中的**潜在**可追回金额，且所有处置需人工确认。
