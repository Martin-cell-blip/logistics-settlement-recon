# 评测、指标与 Bad Case 闭环

## 1. 当前可复现的离线评测

```bash
python src/evaluate_copilot.py
pytest -q
```

当前评测分为三层：

| 层级 | 数据 | 测什么 | 明确不代表什么 |
|---|---|---|---|
| 规则回归 | 21 条非重复合成案件；证据字段与期望标签分别声明 | 裁定与建议动作是否符合规则 | 真实线上准确率、模型质量 |
| 安全兜底 | 10 条证据缺失、状态冲突或金额方向冲突案件 | 是否全部回退 `SUSPECT + 人工复核` | 所有生产 Bad Case 已覆盖 |
| 模型护栏 | 4 个无 API 调用的结构化输出夹具 | 对齐输出放行，裁定/动作冲突和过度置信全部回退 | 某个真实模型效果优秀 |

离线门槛全部为 100%，因为这些测试是发布护栏；任何失败都应阻止合并。真实模型评测必须额外运行：

```bash
python src/evaluate_copilot.py --with-model --model-limit 5
```

该命令会产生实际 API 成本，并记录 `status`、模型/Prompt 版本、P95 时延、输入/输出 Token 和可选成本估算。未运行时，文档和简历不得声称已有模型准确率或模型对比结果。

## 2. 指标分层

| 层级 | 指标 | 定义 | MVP 门槛 |
|---|---|---|---|
| 规则 | 裁定/动作回归率 | 与人工声明夹具一致 | 100% |
| 模型格式 | Schema 合规率 | 输出可被 `ModelReview` 严格解析 | 100% |
| 模型依据 | 证据引用有效率 | 引用 ID 均存在于案件证据账本 | 100% |
| 模型安全 | 规则冲突拦截率 | 改写裁定、动作或抬高置信度时回退 | 100% |
| 业务 | 人工采纳率 | `APPROVED / 已完成决定` | 观察，不预设虚构目标 |
| 效率 | 每案处理时长 | `DECISION_SUBMITTED - CASE_OPENED` | 与人工基线比较 |
| 系统 | P95 时延、Token、单案成本 | API 和模型运行元数据 | 运行中采集 |

必须分别报告规则、模型、体验和业务指标，不能用其中一层代替另一层。

## 3. 上线前独立评测计划

1. 从真实历史账单随机抽样，按异常类型、金额区间和复杂度分层；每类先收集 100–300 条。
2. 两名结算专家独立标注异常类型、可用证据、建议动作和是否必须升级；对分歧裁决并保存标注版本。
3. 开发集仅用于规则、Prompt 和模型调优；冻结测试集只在版本候选确定后运行。
4. 比较规则基线、候选轻量模型和候选增强模型，至少报告 Schema 合规、证据引用、幻觉/冲突、P95、Token 和成本。
5. 上线后把驳回、升级、重复追问和模型回退回流为候选 Bad Case，每周按风险金额与发生频率排序复盘。

## 4. Bad Case 分类与处置

| Bad Case | 风险 | 当前兜底 | 后续优化 |
|---|---|---|---|
| 订单或费用明细缺失 | 证据不足却给出资金建议 | `SUSPECT + 人工复核` | 补数据血缘和缺失字段提示 |
| 异常类型与金额方向冲突 | 上游聚合或标签错误 | 屏蔽资金建议 | 检查币种、税费、舍入与容差 |
| 重复计费与账单行数冲突 | 聚合/主数据错误 | 人工复核 | 排查账单粒度与去重键 |
| 未送达判定与签收状态冲突 | 状态同步延迟 | 人工复核 | 增加状态更新时间和来源 |
| 模型理由与证据不一致 | 幻觉或过度解释 | 只展示规则理由和回退原因 | 扩充 Bad Case 与 Prompt |
| 高金额、低置信度案件 | 错误成本高 | 强制主管升级 | 金额阈值与双人审批 |

## 5. 不夸大的结果表述

可以表述：

> 在合成账单与注入真值上，对账引擎异常查全率约 99.9%；受控复核 MVP 使用 21 条非重复规则回归、10 条证据冲突安全案例和 4 类模型护栏测试验证发布边界。模型仅按需生成结构化证据摘要，所有处置需人工确认。

不可表述：

- “AI 复核准确率 99.9%”；
- “模型自动追回 14 万元”；
- “已经完成真实客户验证”；
- “模型优于人工或规则基线”（在独立标注测试完成前）。

14 万元是模拟账单中的潜在拒付/追回金额，不是已回收资金。
